# Beauty Sales DWH (美容業向け売上分析DWH)

![Java](https://img.shields.io/badge/Java-Spring%20Boot-green) ![Next.js](https://img.shields.io/badge/Next.js-TypeScript-black) ![Docker](https://img.shields.io/badge/Docker-Compose-blue) ![PostgreSQL](https://img.shields.io/badge/DB-PostgreSQL-336791)

美容サロン（ヘア・ネイル・エステ等）向けの**売上分析および顧客データ統合（名寄せ）プラットフォーム**です。
スマレジPOSのAPIからデータを自動収集(ETL)し、DWH構築、名寄せによる顧客ID統合を経て、経営判断に直結するダッシュボードを提供します。

## 📖 プロジェクト背景と課題 (Background)

### プロジェクトの起点
POS データを分析基盤として設計してみたいと思ったことがきっかけです。複数のPOS APIを検討した中で、スマレジAPIがテストデータの準備のしやすさとAPI設計のしやすさで最も開発に向いていたため採用しました。

ドメイン選定の際、20年来の担当美容師さんへのヒアリングを通じて現場の実態を把握しました。「ABC分析は知っているが、活用できているのはITリテラシーの高い一部の美容師のみで、大多数は対前年比を見るだけ」という構造的な課題を確認し、分析の民主化をテーマに設計しました。

中小規模の美容サロンでは、POSシステムで会計を行っているものの、そのデータを**「分析」**に活かせていない課題があります。

* **データの分断:** 分析のためにCSVを出力し、Excelで手作業加工する工数が負担。
* **顧客の重複:** 予約サイト経由や電話予約などで、同一人物が別IDとして登録され、正確なリピート分析やLTV算出ができない。
* **属人的な勘:** 数字に基づいたスタッフ評価やマーケティングができていない。

本プロジェクトは、これらの課題を**「データパイプラインの自動化」**と**「名寄せアルゴリズム」**によって解決する実証的なWebアプリケーションです。

## 📐 アーキテクチャと機能 (Architecture & Features)

### 1. データ収集・統合 (ETL & DWH)
* **スマレジAPI連携:** POSデータ（売上・顧客・スタッフ）を日次バッチで自動取得。
    * POSデータ初回取得時は３ヶ月前までとし、それ以上前のデータを取得したい場合は別途運用(取引明細一覧CSVをスマレジからDLして追加など）で対応とする。
* **Raw / DWH 層の分離:**
    * `Staging Area`: APIレスポンスをそのまま格納。
    * `Data Warehouse`: 分析用にスタースキーマライクに正規化したテーブル群。

* **ID設計:**
システム独自のID付与にはSnowflakeIDを採用。RDBのB-treeインデックス特性上、UUIDv4のような完全ランダム値はページ分割が頻発しパフォーマンスが劣化するため、タイムスタンプベースで単調増加に近いSnowflakeIDを選定し、生成クラスも自前で実装。
* **ドメインモデル設計:**
DDDの考え方に基づきバリデーションをモデルクラスに集約。ビジネスルールをモデルが責任を持つ設計とし、テストも網羅的に実装。

### 2. 顧客名寄せエンジン (Identity Resolution) 🌟Key Feature
本システムの核となる機能です。表記揺れのある顧客データを半自動で統合します。
* 氏名（カナ）、電話番号の一致度から「同一人物候補」を自動検出。
* 管理画面にて、ユーザー（店長）が候補を確認し、IDを統合（Merge）する機能を提供。
* **効果:** 正確な「来店回数」「累計売上」の算出が可能に。

#### 名寄せアルゴリズムの設計思想
同一人物の検出には確率論的アプローチを採用し、複数の一致条件にスコアを付与して判定しています。
| スコア| 条件 | 
| :--- | :--- | 
|100点|電話番号一致 かつ カナ名字（先頭2文字）一致|
|95点|カナ完全一致|
|90点|漢字完全一致|
|85点|漢字曖昧一致（Levenshtein距離 ≤ 1）|
|80点|カナ曖昧一致（Levenshtein距離 ≤ 1）|
|80点|電話番号一致（名字不一致・世帯別の可能性）|

#### 実装上の工夫
* 正規化処理: 全角・半角スペース除去、電話番号の数字以外を除去してから比較
* ブロッキング: 全件総当たりを避けるため、カナ先頭2文字または電話番号が一致するペアのみに絞り込んでから類似度計算
* Levenshtein距離: ostgreSQLのfuzzystrmatch拡張のlevenshtein関数を活用
* マージ優先度: 来店回数・直近来店日をもとにtarget/sourceを自動決定
* AIを活用してテストコード・テストデータを生成し、ポジティブテスト・ネガティブテストの両面から検出精度と誤検出を担保しています。生成物は全て自分でレビューし、内容を把握した上で採用しています。

### 3. 分析ダッシュボード
* **月次売上推移:** 昨対比を含めた棒・折れ線複合グラフ。
* **スタッフ別ランキング:** 売上貢献度を可視化。
* **ABC分析 (パレート図):** 上位顧客（ロイヤルカスタマー）の特定。
